# 1. Terraform State File

One of the most important Terraform concepts.

---

# Definition

Terraform state file maintains infrastructure mapping between:
- Terraform configuration
- Actual cloud infrastructure

---

# Default State File

```bash
terraform.tfstate
```

---

# Responsibilities of State File

| Responsibility | Description |
|---|---|
| Resource Tracking | Tracks created resources |
| Metadata Storage | Stores infrastructure metadata |
| Dependency Mapping | Maintains resource relationships |
| Infrastructure Mapping | Maps Terraform to real cloud resources |

---

# Important Engineering Concept

```text
Terraform state file acts as the source of truth for infrastructure management.
```

---

# Important Interview Point

```text
Terraform uses the state file to understand what infrastructure already exists.
```

---

# 2. Remote Backend Architecture

Production environments never rely on local state storage.

---

# Why Local State is Risky

| Problem | Impact |
|---|---|
| Laptop failure | State loss |
| No collaboration | Team conflicts |
| Security risk | Sensitive data exposure |
| No locking | State corruption |

---

# Solution

```text
Remote Backend
```

---

# Azure Remote Backend Architecture

```text
Terraform
      ↓
Azure Storage Account
      ↓
Blob Container
      ↓
terraform.tfstate
```

---

# Benefits of Remote Backend

| Benefit | Description |
|---|---|
| Team Collaboration | Shared state management |
| State Locking | Prevent concurrent corruption |
| Centralized Storage | Easier management |
| Backup & Recovery | Durable storage |
| Security | Controlled access |

---

# Important Interview Point

```text
Remote backend enables centralized, secure and collaborative Terraform state management.
```

# 3. Terraform State Locking

Critical production-grade Terraform concept.

---

# Problem Scenario

Two engineers running:

```bash
terraform apply
```

at the same time.

---

# Risk

```text
State corruption
```

---

# Solution

```text
Terraform State Locking
```

---

# Azure Backend Mechanism

Azure Blob Storage uses:
- Blob lease locking mechanism

---

# Workflow

```text
Engineer A → Acquire Lock
          ↓
terraform apply running
          ↓
Engineer B waits
          ↓
Lock released
```

---

# Important Interview Point

```text
Terraform state locking prevents simultaneous infrastructure modifications from corrupting the state file.
```

# 4. Infrastructure Drift

One of the most asked Terraform interview topics.

---

# Definition

Infrastructure drift occurs when actual cloud infrastructure differs from Terraform-managed configuration or state.

---

# Real-Time Examples

| Scenario | Result |
|---|---|
| VM deleted from portal | Drift |
| NSG manually modified | Drift |
| Storage settings changed manually | Drift |

---

# Engineering Meaning

Terraform expects:
```text
Code = State = Cloud
```

When mismatch occurs:
```text
Drift happens
```

---

# Important Interview Point

```text
Infrastructure drift occurs when resources are modified outside Terraform control.
```

---

# 5. Drift Detection

Terraform automatically detects infrastructure drift.

---

# Detection Command

```bash
terraform plan
```

---

# Internal Terraform Logic

Terraform compares:
- Terraform code
- State file
- Actual cloud infrastructure

---

# Detection Flow

```text
Terraform Code
       ↓
State File
       ↓
Azure Cloud APIs
       ↓
Mismatch Detection
```

---

# Important Interview Point

```text
terraform plan identifies infrastructure drift by comparing desired and actual infrastructure states.
```

# 7. count vs for_each

Very important Terraform interview comparison.

---

# count

Uses:
- numeric indexing

---

# Example

```hcl
count = 3
```

---

# Problems with count

| Problem | Description |
|---|---|
| Index shifting | Resource remapping risk |
| Reordering issues | Infrastructure instability |

---

# for_each

Uses:
- unique keys
- stable mapping

---

# Example

```hcl
for_each = toset(["dev", "prod"])
```

---

# Benefits of for_each

| Benefit | Description |
|---|---|
| Stable mapping | Better tracking |
| Safer updates | Less recreation |
| Production preferred | Better scalability |

---

# Important Interview Point

```text
for_each is preferred over count because it provides stable resource mapping using unique keys.
```
